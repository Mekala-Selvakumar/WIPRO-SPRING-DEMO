The BeanFactory interface provides an advanced configuration mechanism capable of managing any type of object. ApplicationContext is a sub-interface of BeanFactory. It adds:

Easier integration with Spring’s AOP features

Message resource handling (for use in internationalization)

Event publication

Application-layer specific contexts such as the WebApplicationContext for use in web applications.

In short, the BeanFactory provides the configuration framework and basic functionality, and the ApplicationContext adds more enterprise-specific functionality. The ApplicationContext is a complete superset of the BeanFactory and is used exclusively in this chapter in descriptions of Spring’s IoC container. 


The org.springframework.context.ApplicationContext interface represents the Spring IoC container and is responsible for instantiating, configuring, and assembling the beans

The container gets its instructions on the components to instantiate, configure, and assemble by reading configuration metadata. 
The configuration metadata can be represented as annotated component classes, configuration classes with factory methods, or external XML files or Groovy scripts.


Configuration Metadata can be 
 1. XML  based
 2. Java based
3. Annotation based

With either format, you may compose your application and the rich interdependencies between those components.

In Spring Core (XML configuration), init-method  and destroy-method  are used to perform lifecycle callbacks on a bean — that is, custom actions you want Spring to run  immediately after a bean is created  and just before it is destroyed
 
1. init-method — Runs After Bean Creation

This is used for initialization logic

Spring will call this method after:

1. The object is created
2. Dependencies are injected
3. Before the bean is ready for use

 Example use cases:

* Opening a database connection
* Loading a file / cache
* Validating configurations
* Starting background processes

XML Example

 
<bean id="myBean" class="com.example.MyBean" init-method="init">
</bean>

Corresponding Java Class
 
public class MyBean {
    public void init() {
        System.out.println("Init method called - resources initialized.");
    }
}
 

2. `destroy-method` — Runs Before Bean Destruction

This method is used to perform cleanup logic.
Spring will call this method just before the bean is destroyed (when the container is shutting down).

 Example use cases:

* Closing database connections
* Releasing file handles
* Stopping background threads
* Cleaning up caches

XML Example

 
<bean id="myBean" class="com.example.MyBean" destroy-method="cleanup">
</bean>
 
Corresponding Java Class

 
public class MyBean {
    public void cleanup() {
        System.out.println("Destroy method called - resources cleaned.");
    }
}
 

 Important: Destroy method works only for:

* Singleton–scoped beans
* And when you close the container manually:

 
((ClassPathXmlApplicationContext) context).close();
 

 

 Complete Example

XML:
 
<bean id="dbConnection" class="com.example.DBConnection"
      init-method="openConnection" 
      destroy-method="closeConnection">
</bean>
 

Class:

 
public class DBConnection {

    public void openConnection() {
        System.out.println("DB Connection opened.");
    }

    public void closeConnection() {
        System.out.println("DB Connection closed.");
    }
}

No default constructor is present.
Spring will use the parameterized constructor automatically.
This is called constructor injection.

From Spring 4.3 onward, if there is only one constructor, @Autowired is optional.

@Component
public class EmployeeService {

    private EmployeeRepository repo;

    @Autowired
    public EmployeeService(EmployeeRepository repo) {
        this.repo = repo;
    }
}


@Component
public class EmployeeService {

    private EmployeeRepository repo;

    public EmployeeService(EmployeeRepository repo) {
        this.repo = repo;
    }
}

Both are valid.



public Customer(int customerId, String customerName, String emailId) {

    this.customerId = customerId;
    this.customerName = customerName;
    this.emailId = emailId;

    System.out.println("Parameterized Constructor : " + customerId);
}

If you annotate the class with @Component:
@Component
public class Customer {


Spring will NOT be able to create the object automatically because:

int
String

are not Spring beans by default.

Spring knows how to inject objects like:

@Autowired
private CustomerService service;


because CustomerService is a bean.

But for primitive/simple values, Spring needs explicit values.

You can make it work using @Value:

@Component
public class Customer {

    private int customerId;
    private String customerName;
    private String emailId;

    public Customer(
            @Value("101") int customerId,
            @Value("John") String customerName,
            @Value("john@gmail.com") String emailId) {

        this.customerId = customerId;
        this.customerName = customerName;
        this.emailId = emailId;

        System.out.println("Parameterized Constructor : " + customerId);
    }
}

Spring can automatically detect stereotyped classes and register corresponding BeanDefinition instances with the ApplicationContext. For example, the following two classes are eligible for such autodetection:

 
@Component
public class MovieFinder {

    public void findMovie() {

        System.out.println("Movie Found");
    }
}

@Service
public class SimpleMovieLister {

	private final MovieFinder movieFinder;

	public SimpleMovieLister(MovieFinder movieFinder) {
		this.movieFinder = movieFinder;
	}
}
